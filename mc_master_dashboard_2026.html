⚠ COMPLIANCE ALERT — Treat Every Contact Deletion Case as Urgent

Contact deletion failures are not just support issues — they are potential GDPR/CCPA violations. Any case where deletion is confirmed to have failed (not just slow) must be escalated immediately regardless of stated severity.

🔑 Critical Facts Every New Hire Must Know

1. Contact Deletion API = only method to permanently remove PII from MC
2. It only applies to sendable DEs — non-sendable DEs require a custom SSJS script
3. Deletion is the lowest-priority background process — yields to sends, journeys, imports, automations
4. A single contact deletion cascades across: All Subscribers, MobileConnect, MobilePush, DEs, system tables
5. Deletion is permanent — no data retention, no history
6. A 200 OK response means request was accepted, not completed — deletion is async
7. Concurrent sync + deletion on same DE can cause apparent failures (lock conflict)

🔑 Wait by Attribute — Key Behaviors Every New Hire Must Know

1. If attribute date field is NULL when contact enters Wait → contact is released immediately the next day at 8 AM EST (not an error — expected behavior)
2. Wait queue is FIFO and shared across the entire database — high volume injections cause delays for ALL journeys on that DB
3. "Extend wait until specific time" sets the expireTime — contacts won't move before that exact UTC time even if attribute condition is met
4. Re-entry Anytime + Wait by Attribute = same contact can be in Wait multiple times simultaneously
5. Journey Data vs Live Data: use Journey Data to snapshot the attribute at entry time; use live data if value may update after entry
6. Wait queue capacity: ~2M contacts released/hour — large injections (6M+) cause visible delays
